# FSM Declaration Name Conflict Resolution

## Context

FSM modules share a flat declaration namespace with host apps. A module declaring `query "getTodos"` conflicts with any app or other module using the same name. Two distinct problems arise:

1. **Declaration name collision** - `App.use()` currently throws on duplicates, making any common name (`getUser`, `MainRoute`, `HomePage`) hazardous for module authors.
2. **SDK symbol collision** - Operation names flow directly to `export const getTodos: ...` in `wasp/client/operations`. If we fix (1) by auto-prefixing, module client code that imports `getTodos` breaks since the SDK now exports it as `waspelloTodoModule_getTodos`.

## Solution overview

**Problem 1:** Auto-prefix all module declaration names in `App.use()` before merging into the flat namespace. Package `@waspello/todo-module` → prefix `waspelloTodoModule` → declaration `getTodos` → `waspelloTodoModule_getTodos`.

**Problem 2:** Module client code uses `@pkgName/operations` (a package export path) instead of `wasp/client/operations`. This resolves to different things in dev mode vs host app mode, always giving clean unqualified names.

## Part 1: Auto-prefix in `App.use()`

**File:** `waspc/data/packages/wasp-config/src/publicApi/App.ts`

### 1.1 Package prefix derivation

Add a private static method:

```typescript
static #packagePrefix(packageName: string): string {
  // "@waspello/todo-module" → "waspelloTodoModule"
  // "@myorg/payments"       → "myorgPayments"
  // "plain-pkg"             → "plainPkg"
  return packageName
    .replace(/^@/, '')
    .replace(/[/-](.)/g, (_, c: string) => c.toUpperCase())
    .replace(/[^a-zA-Z0-9]/g, '');
}
```

### 1.2 Prefix all declaration names

Add a private static method `#prefixModuleDeclarations(spec: TsModuleSpec, prefix: string): void`:

1. Collect current page names: `const originalPageNames = new Set(spec.pages.keys())`
2. For each of the 8 `mapKeys` (actions, apiNamespaces, apis, cruds, jobs, pages, queries, routes): re-key every entry to `prefix + "_" + name`
3. After re-keying, iterate the new `routes` map and for each route where `routeConfig.to` is in `originalPageNames`, update `to` to `(prefix + "_" + routeConfig.to) as PageName`

Route cross-references must be fixed because `RouteConfig.to` is a `PageName` (branded string) — a module route references a module page by name, and both names must be updated consistently.

### 1.3 Wire into `use()`

After `#rewriteModuleImports`, add:
```typescript
if (packageName && packageName !== App.projectPackageName) {
  App.#prefixModuleDeclarations(incoming, App.#packagePrefix(packageName));
}
```

The duplicate-name `throw` stays as a safety net but becomes unreachable for properly named external modules.

### 1.4 Track original operation names for SDK generation

Before prefixing, record the original query/action names from the module:

Add `moduleOperationMaps: ModuleOperationMapEntry[]` to `TsAppSpec` (in `tsAppSpec.ts`):
```typescript
type ModuleOperationMapEntry = {
  packageName: string;
  queryNames: string[];   // original (unprefixed) names
  actionNames: string[];  // original (unprefixed) names
};
```

Populate this in `use()` for each external module (before prefixing) and expose via `GET_TS_APP_SPEC`.

### 1.5 Update tests

**File:** `waspc/data/packages/wasp-config/__tests__/module.unit.test.ts`

- Update duplicate-detection tests: named module vs app no longer throws — the module's name gets prefixed, so both coexist. Keep error tests for anonymous modules only.
- Add tests for `#packagePrefix` (various package name formats).
- Add tests verifying prefixed keys in merged maps.
- Add tests verifying `route.to` is rewritten after prefix.
- Add tests verifying two modules with identical declaration names both merge without error.

## Part 2: Module-scoped SDK operations

Module client code imports from `@pkgName/operations`, not `wasp/client/operations`:

```typescript
// In module's React component
import { getTodos, createTodo } from '@waspello/todo-module/operations';
```

### 2.1 Dev mode (standalone module)

Module's `package.json` already has `./*` export glob. Add a `dist/operations.js` entry point generated by `wasp module build`:

**File:** `waspc/src/Wasp/Generator/ModuleGenerator/SdkGenerator.hs` — add generation of `dist/operations.js`:
```javascript
// Auto-generated by wasp module build
export * from './client/operations/index.js';
```
And `dist/operations.d.ts` for TypeScript consumers. This re-exports from the module's own standalone SDK where operations have clean names.

### 2.2 Host app mode — per-module generated SDK file

**File:** `waspc/src/Wasp/Generator/SdkGenerator/Client/OperationsGenerator.hs`

Add `genModuleOperationsFiles :: AppSpec -> Generator [FileDraft]`. For each entry in `moduleOperationMaps`, generate `sdk/wasp/client/operations/modules/@scope/pkg-name.ts`:

```typescript
// Generated: sdk/wasp/client/operations/modules/@waspello/todo-module.ts
import { type QueryFor, createQuery } from '../queries/core';
import type { waspelloTodoModule_getTodos as GetTodos_fn } from 'wasp/server/operations';

// Exports CLEAN original names, uses prefixed HTTP route internally
export const getTodos: QueryFor<GetTodos_fn> = createQuery(
  'operations/waspello-todo-module-get-todos',  // camelToKebabCase of prefixed name
  [],
);
```

The HTTP route string is `camelToKebabCase(prefix + "_" + originalName)`, which matches what the server generates for the prefixed AppSpec operation name.

### 2.3 Host app mode — Vite alias

**File:** wherever Vite config is generated (likely `waspc/src/Wasp/Generator/WebAppGenerator/` or the `waspConfig.ts` template)

For each module with operations, inject a `resolve.alias` entry:
```typescript
{
  find: '@waspello/todo-module/operations',
  replacement: path.resolve('.wasp/out/sdk/wasp/client/operations/modules/@waspello/todo-module.ts'),
}
```

This makes the module's `import { getTodos } from '@waspello/todo-module/operations'` resolve to the generated SDK file with clean names.

### 2.4 Pass `moduleOperationMaps` through to Haskell

**File:** `waspc/data/packages/wasp-config/src/mapTsAppSpecToAppSpecDecls.ts` — serialize `moduleOperationMaps` into the JSON output.

**File:** `waspc/src/Wasp/AppSpec/App.hs` — add `ModuleOperationMap` type and field to `App`.

## Edge cases

- **Modules without operations:** `#prefixModuleDeclarations` still runs (prefixing routes/pages). No per-module SDK file is generated. No Vite alias added.
- **Module IS the host app package:** `packageName === App.projectPackageName` guard skips prefixing. Already handled.
- **Anonymous modules:** No prefix applied (no `packageName`). Error-on-duplicate still applies.
- **CRUDs:** Declaration name gets prefixed. Generated CRUD HTTP paths follow. Client-side `wasp/client/crud` would export the ugly prefixed name — a `@pkgName/crud` subpath is needed too, but out of scope for now (same pattern as operations).
- **Prefix collision:** e.g. `@foo/bar-baz` + `thing` vs `@foo/bar` + `bazThing` → same prefix. Very unlikely; treat as a known limitation for now.

2. Check the merged AppSpec has `waspelloTodoModule_getTodos` and `otherModule_getTodos` as separate keys.
3. Check route `to` fields are updated correctly when a module declares both a page and a route referencing it.
4. Build a host app using both modules — no TypeScript duplicate-export errors in `wasp/client/operations`.
5. In a module's React component, `import { getTodos } from '@waspello/todo-module/operations'` resolves correctly in both dev and host-app mode with the right type.
